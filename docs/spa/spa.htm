SPA

Single Page Web Applications

A multi-page web app (MPA) replaces the entire page with new content when the user navigates the browser to another page. A single-page web application only replaces the content that changes with XmlHttpRequest or fetch instructions. During the entire session, as long as the user stays on the website, the website homepage (index.htm) is not replaced or reloaded. This is the SPA host document, which provides the "static" portions, such as header, footer, menus and site navigation. Only the content portions, which correspond to the content of traditional web pages, is replaced when the user navigates within the website.

The Navigation object, available in Edge and Chrome browsers at the time of writing, makes it possible. In browsers where the Navigation object is not available, it is harder to achieve similar results.

Content documents are retrieved dynamically with the XMLHTTPRequest object or Promise-based fetch() instructions.

The WebCAT spa-host and spa-content components encapsulate the Navigation object and provide necessary scripts and styles.

MPA/SPA Coexistence

WebCAT supports multiple SPA host documents in a website project, as well as a mixture of "traditional" MPA and SPA pages.

Transition from an MPA to an SPA page works seamlessly. The redirection script in the SPA page redirects the browser to the SPA host page and provides the content document address in the URL search parameters (in the GOTO parameter). Similarly, a link from an SPA page to an MPA document navigates the browser to the MPA document. Since the MPA documents are self-contained, everything is fine.

Content Containers

Several elements can be used for a content document container: DIV, MAIN, IFRAME or OBJECT, among others. While the simple container elements such as DIV and MAIN integrate the content as integral part of the host document, OBJECT and IFRAME provide some kind of isolation between content and host documents.

If host and content documents have the same origin, the isolation barrier is not very high, code in the host document can easily access elements in the content document, and vice versa.

Script elements in work as usual in OBJECT and IFRAME containers, but they need a "recreation treatment" if they are injected into one of the simple containers. Similarly, stylesheets must be recreated when injected into simple containers.

Relative URLs in content documents pose a problem when they are injected into simple containers. The address base changes from the content documemt location to the host document location. This also requires some special treatment.

The web component technology provides another option to inject content into the host document.

Scrolling to Target Elements

A hash in the URL should load the target document and then should scroll to the hash target element. Unfortunately, it doesn't work for the IFRAME in the SPA host document. The browser scrolls the host document to the hash target element as soon as handler() returns control to its caller, but at this point in time the content document has not yet been loaded.

At the end of the document init code, the target is scrolled into view. Note that this is done in a requestAnimationFrame handler, because it takes some time to render the document to the screen. Two consecutive requestAnimationFrame() turned out to be reliable.

A beneficial side effect is that scrolling is now "smooth".

The scroll margins need to be set according to sticky toolbars at the top and/or bottom of the window. They prevent that sticky toolbars obscure the link target.

Things are different when the content document is already loaded and the user navigates to a link target element on that page. In this case, NavigateEvent.hashChange===true. This means that the host document should NOT reload the content document (it's already there), and instead just scroll to the target element. This is the reason why the event handler returns if that condition is fulfilled.
