Publishing Changes

Method 1: XCopy

cd ...\inc\docs\webcat
xcopy /d /s ..\..\..\web-toolbox\code\folder-name folder-name\

The /u option copies only files that already exist in the target directory.

The /d option causes only newer files to be copied.

Directory Hardlinks (Junctions)

To publish a WTX2 component, create a directory hardlink (junction):

cd ...\inc\docs\webcat
mklink /j folder-name ..\..\..\web-toolbox\code\folder-name

Changes in the web-toolbox\code\folder-name are automatically reflected in the corresponding global include folder.

Using hardlinks can lead to unexpected loss of data, especially if you work with multiple junctions to the same folder! If the last junction to a folder is removed, the files and subfolders become inaccessible. Windows Explorer does not issue a warning.

Using the DOS del folder-name command always deletes the files in that folder, regardless how many junctions exist that point to that folder.

The DOS rmdir command deletes a junction without warning, except it it was the last junction. In this case, you get the error message "folder not empty". So the rmdir command is considered to be safe.

File Copy

Changes are usually made in the toolbox code folder, then the files are copied to the public include folder. Forgetting to copy the file is easily noticed because changes don't appear in the documents.

Editing files in the global include folder is also possible. Changes immediately appear in the HTML documents. But the developer must remember to copy modified files to the development code folders. This can easily be forgotten…

Merge

A GUI merge tool (WinMerge for example) is perfect for a quick visual check for differences between the global include and the development code folder(s). They can compare entire folder trees and move the changes back an forth.

Directory Change Watcher

A directory change watcher can automate copying modified files from development code folders to the global include folder. There is a C/C++ project in the pipeline.
